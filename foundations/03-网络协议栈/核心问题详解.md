# 网络协议栈核心问题详解

针对四个核心问题的深入解释。

---

## 1. TCP 三次握手：为什么不是两次？

### 核心问题

两次握手的问题：**无法防止历史重复连接请求**。

### 具体场景

```
时间线：

T1: 客户端发送 SYN (seq=100)
    ↓
    网络延迟，SYN 滞留
    
T2: 客户端没收到响应，重发 SYN (seq=200)
    ↓
    服务器收到，回复 SYN-ACK (seq=500, ack=201)
    ↓
    客户端回复 ACK (ack=501)
    ↓
    连接建立，数据传输，连接关闭

T3: 滞留的 SYN (seq=100) 突然到达服务器
    ↓
    【如果是两次握手】
    服务器回复 SYN-ACK (seq=600, ack=101)
    ↓
    连接建立！（但客户端早已关闭）
    ↓
    服务器一直等客户端发数据，资源浪费
    
    【三次握手】
    服务器回复 SYN-ACK (seq=600, ack=101)
    ↓
    客户端收到：这是什么？我已经关闭了啊
    ↓
    客户端发送 RST（复位），连接终止
    ↓
    服务器释放资源
```

### 另一个角度：同步初始序号

```
两次握手只能同步一方的序号：

客户端：我的初始序号是 100
服务器：知道了（但服务器自己的序号呢？）

三次握手同步双方序号：

客户端：我的初始序号是 100（SYN）
服务器：我的初始序号是 500，期待你的 101（SYN-ACK）
客户端：知道了，期待你的 501（ACK）

结果：双方都确认了对方的初始序号
```

## 2. 滑动窗口：流量控制 vs 拥塞控制

### 两个不同的问题

| | 流量控制（Flow Control） | 拥塞控制（Congestion Control）|
|---|------------------------|---------------------------|
| **解决什么问题** | 接收方处理不过来 | 网络传输不过来 |
| **控制对象** | 发送方 vs 接收方 | 发送方 vs 网络 |
| **反馈来源** | 接收方通告窗口（rwnd） | 发送方探测（丢包、延迟）|
| **目的** | 防止压垮接收方 | 防止压垮网络 |

### 流量控制：接收方缓存

```
接收方缓存状态：
┌─────────────────────────────────────────┐
│ 已确认并交付应用 │ 已接收未确认 │ 空闲  │
│    1-1000      │  1001-3000  │ 3001+ │
│                │             │       │
│                │  ← 这部分占用缓存    │
└─────────────────────────────────────────┘

接收方在 TCP 首部中设置 Window = 2000
表示：我还能接收 2000 字节

发送方据此调整：
已发送未确认 ≤ Window
```

### 拥塞控制：网络容量

```
发送方维护两个窗口：

有效窗口 = min(rwnd, cwnd)
         ↑         ↑
       流量控制  拥塞控制

拥塞窗口 cwnd 的调整：

慢启动阶段（探测网络容量）：
cwnd = 1 MSS
每收到一个 ACK: cwnd += 1 MSS
每 RTT: cwnd 翻倍

拥塞避免阶段（接近容量）：
当 cwnd > ssthresh:
每 RTT: cwnd += 1 MSS（线性增长）

丢包时（网络拥塞）：
快速重传（3 个重复 ACK）:
  ssthresh = cwnd / 2
  cwnd = ssthresh + 3 MSS
  
超时重传（RTO 到期）:
  ssthresh = cwnd / 2
  cwnd = 1 MSS（重新开始慢启动）
```

### 形象对比

```
流量控制：
你（发送方）→ 朋友家（接收方）
朋友："我家只能放 10 个包裹"
你：只发 10 个，等他说"又腾出地方了"再发

拥塞控制：
你 → 快递公司 → 朋友
发现快递积压（丢包）："最近快递太慢了，少发点"
发现快递顺畅："可以多寄点了"
```

## 3. 序号和确认号：保证顺序和可靠性

### 字节流编号

```
发送数据："HelloWorld"（10 字节）

每个字节一个序号：
H(1) e(2) l(3) l(4) o(5) W(6) o(7) r(8) l(9) d(10)

分成两个包发送：
包 1: seq=1, 数据 "Hello"（字节 1-5）
包 2: seq=6, 数据 "World"（字节 6-10）
```

### 确认机制

```
接收方收到包 1（seq=1, 数据 1-5）：
回复 ACK: ack=6
含义：字节 1-5 已收到，期待字节 6

接收方收到包 2（seq=6, 数据 6-10）：
回复 ACK: ack=11
含义：字节 6-10 已收到，期待字节 11（下一个）

发送方收到 ACK=6：
知道 1-5 已确认，可以从窗口中移除
继续发送后续数据
```

### 乱序处理

```
网络导致包 2 先到达：

接收方：
收到包 2（seq=6），但期待 seq=1
┌─────────────────────────────────────────┐
│ 空洞：1-5 缺失 │ 已收到：6-10 │         │
└─────────────────────────────────────────┘

处理：
1. 暂存包 2（不能交付给应用，前面有缺失）
2. 回复 ACK=1（表示还在期待 1）

收到包 1（seq=1）：
┌─────────────────────────────────────────┐
│ 已收到：1-5 │ 已收到：6-10 │            │
└─────────────────────────────────────────┘

现在连续了，按序交付给应用
回复 ACK=11
```

### 重复检测

```
包 1 丢失，发送方超时重传：

T1: 发送包 1
T2: 没收到 ACK，重发包 1
T3: 第一个包 1 突然到达（延迟）
T4: 重传的包 1 到达

接收方：
收到第一个包 1：正常处理，回复 ACK
收到第二个包 1：序号 1-5 已经确认过了
     ↓
     丢弃重复包
     再回复一次 ACK（防止 ACK 丢失）
```

## 4. 分层：各层如何协作

### 数据封装流程

```
应用层：用户数据 "Hello"
        ↓
传输层（TCP）：
  加 TCP 首部
  ┌─────────────────────────────────────────┐
  │ 源端口 │ 目的端口 │ 序号 │ 确认号 │ ... │
  │  54321 │   80    │ 100  │  0    │     │
  └─────────────────────────────────────────┘
  │              数据："Hello"              │
  └─────────────────────────────────────────┘
        ↓
网络层（IP）：
  加 IP 首部
  ┌─────────────────────────────────────────┐
  │ 版本 │ 头长 │ 总长度 │ TTL │ 协议 │ ... │
  │  4   │  5   │   45   │ 64  │ TCP │     │
  └─────────────────────────────────────────┘
  │ 源 IP：192.168.1.100 │ 目的 IP：93.184.216.34 │
  └─────────────────────────────────────────┘
  │              TCP 段（含数据）            │
  └─────────────────────────────────────────┘
        ↓
链路层（以太网）：
  加以太网首部和尾部
  ┌─────────────────────────────────────────┐
  │ 目的 MAC │ 源 MAC │ 类型 │              │
  │ AA:BB...│CC:DD...│ 0x0800│             │
  └─────────────────────────────────────────┘
  │              IP 数据报                   │
  └─────────────────────────────────────────┘
  │ CRC 校验 │
  └──────────┘
        ↓
物理层：转换为电信号发送
```

### 各层独立演进

```
应用层升级（HTTP/1.1 → HTTP/2）：
- 传输层无感知，继续提供 TCP 服务
- 网络层无感知，继续传输 IP 包

传输层升级（TCP 拥塞控制算法改进）：
- 应用层无感知，继续调用 socket API
- 网络层无感知，继续封装 IP 首部

网络层升级（IPv4 → IPv6）：
- 应用层无感知（通过 DNS 解析不同地址）
- 传输层无感知（TCP 首部不变）
- 链路层无感知（继续封装成帧）

链路层升级（以太网 100M → 1G → 10G）：
- 上层全部无感知，速度自动提升
```

### 分层的好处

```
1. 简化设计
   每层只解决一个问题
   
2. 独立实现
   不同厂商实现同一层接口即可互通
   
3. 灵活替换
   应用层：HTTP 可以换成 FTP、SMTP
   传输层：TCP 可以换成 UDP
   链路层：以太网可以换成 WiFi、4G
   
4. 易于调试
   问题分层定位：
   - ping 不通 → 网络层问题
   - 能 ping 通但网页打不开 → 传输层/应用层问题
```

## 一句话总结

| 问题 | 核心机制 | 关键理解 |
|-----|---------|---------|
| 三次握手 | 防止历史连接 + 同步序号 | 第三次让客户端有机会拒绝错误连接 |
| 流量/拥塞控制 | rwnd vs cwnd | 一个防接收方过载，一个防网络过载 |
| 序号/确认号 | 字节流编号 + 累积确认 | 乱序包暂存，连续后交付 |
| 分层 | 封装 + 解封装 | 每层加/去首部，独立演进 |
