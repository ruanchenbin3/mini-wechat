# 网络协议栈：两台电脑怎么"打电话"

核心问题：两台电脑之间传数据，会遇到哪些坑？

---

## 场景一：直接喊话的问题

### 例子：你在教室喊"中午吃啥"

```
你（北京）："中午吃啥"（大声喊）
           ↓
问题1：噪音干扰
同学 A 听到："中午吃*啥"（某个字没听清）
           ↓
问题2：多人同时喊
同学 B 也在喊："下午打球"
你们的声音混在一起，谁都听不清
           ↓
问题3：对方不在
同学 C 出去上厕所了，没听到
你一直在等回应，不知道他听没听到
           ↓
问题4：顺序乱了
你先问"吃啥"，再问"去哪吃"
对方先听到"去哪吃"，再听到"吃啥"
完全理解反了
```

**直接通信的四个问题**：
1. **出错**（听不清）
2. **冲突**（多人同时说）
3. **丢失**（对方没收到）
4. **乱序**（顺序错了）

## 场景二：快递送货的问题

### 例子：你给上海的朋友寄一本书

```
传统快递（无协议）：

你：把书扔出窗外，希望它能到上海
   ↓
书：掉在地上/被狗叼走/被风吹到河里
   ↓
朋友：永远没收到，你也不知道

问题：
- 没有地址，不知道往哪送
- 没有确认，不知道到没到
- 没有包装，书可能损坏
- 没有顺序，多本书可能先到后寄的
```

**现实快递如何解决**：
1. **地址**（省市区街道门牌号）
2. **单号**（追踪物流状态）
3. **包装**（防水防震）
4. **签收**（确认收到）

**网络协议就是电脑的"快递系统"**。

## 分层：快递公司的组织架构

为什么快递公司要分部门？

```
┌─────────────────────────────────────┐
│ 客服部（应用层）                     │
│ "客户要寄一本书到上海"               │
│ 不关心怎么运输，只关心需求           │
├─────────────────────────────────────┤
│ 分拣部（传输层）                     │
│ "确保书完整送达，丢件要补发"         │
│ 打包、贴单号、追踪、签收             │
├─────────────────────────────────────┤
│ 运输部（网络层）                     │
│ "找到去上海的路"                     │
│ 北京→天津→济南→南京→上海           │
│ 路堵了换条路                         │
├─────────────────────────────────────┤
│ 车队（链路层）                       │
│ "把货送到下一站"                     │
│ 卡车/飞机/轮船，不管包里是什么       │
├─────────────────────────────────────┤
│ 司机（物理层）                       │
│ "开车、开飞机"                       │
│ 只关心怎么移动物理货物               │
└─────────────────────────────────────┘

每层只和上下层打交道，不关心其他层
客服不知道卡车走哪条路
司机不知道运的是什么书
```

## TCP：确保书完整送达

### 问题1：书在运输中损坏（出错）

```
现实快递：
收货时开箱检查，损坏拒收，要求重发

TCP 解决：校验和（Checksum）

发送方：
数据："Hello"
计算校验和：0x1A2B
一起发送：[数据"Hello"][校验和0x1A2B]

接收方：
收到数据，重新计算校验和：0x1A2B
对比：一致 → 没问题
      不一致 → 数据损坏，丢弃，要求重发
```

### 问题2：书丢了（丢失）

```
现实快递：
单号追踪，显示"已签收"才算完成
长时间没更新，打电话问，丢了补发

TCP 解决：确认应答（ACK）+ 超时重传

发送方：发送数据包 #1
         ↓
         等待...
         ↓
接收方：收到 #1，回复 "ACK #2"（表示#1收到，期待#2）
         ↓
发送方：收到 ACK，继续发 #2

如果超时没收到 ACK：
发送方：重发 #1
         ↓
接收方：收到重复的 #1，丢弃，再发一次 ACK
         （防止 ACK 本身丢失导致发送方重发）
```

### 问题3：朋友家太小，堆不下太多货（流量控制）

```
场景：
你：一口气发 100 本书
朋友家：只有 10 平米，堆了 50 本就满了
结果：后面的书只能堆门外，被雨淋湿（丢失）

TCP 解决：滑动窗口

朋友："我家还能放 20 本书"（窗口大小=20）
你：只发 20 本，等朋友说"又腾出地方了"再发

动态调整：
- 朋友家大 → 多发几本
- 朋友家小 → 少发几本
- 永远不压垮对方
```

### 问题4：高速公路堵车（拥塞控制）

```
场景：
双 11，所有人同时发货
高速公路堵死，快递车寸步难行
你还拼命发货，越堵越严重

TCP 解决：慢启动 + 拥塞避免

开始：发 1 本书（试探）
      ↓
顺利：发 2 本 → 4 本 → 8 本（指数增长）
      ↓
发现丢包（堵车了）：
      窗口减半（8本→4本）
      线性增长（4本→5本→6本）
      慢慢接近极限，但不突破
      ↓
又丢包：再减半，再线性增长

结果：自动找到"刚好不堵"的速度
```

### 问题5：多本书顺序乱了

```
场景：
你寄了 3 本书：《1》《2》《3》
快递路线不同：
  《2》走了空运，先到
  《1》走了陆运，后到
  《3》丢了，重发，最后到

朋友收到：《2》《1》《3》
结果：先看《2》完全看不懂（依赖《1》的内容）

TCP 解决：序号（Sequence Number）

每本书贴标签：
  《1》标签：第 1-100 页
  《2》标签：第 101-200 页
  《3》标签：第 201-300 页

朋友收到后按标签排序：
  收到《2》（101-200）→ 先存着，等《1》
  收到《1》（1-100）→ 可以读《1》了
  收到《3》（201-300）→ 按顺序读

结果：无论先到后到，最终顺序正确
```

## 三次握手：建立连接

### 例子：打电话

```
你打给女朋友：

你："喂，能听到吗？"（SYN）
    ↓
女朋友："能听到，你能听到我吗？"（SYN-ACK）
    ↓
你："能听到！"（ACK）
    ↓
开始聊天

为什么不能两次？

两次握手的问题：
你："喂，能听到吗？"（网络延迟，没送到）
你："喂，能听到吗？"（重发，送到了）
女朋友："能听到！"（两次握手建立连接）
你们聊天，结束，挂断

延迟的那条消息突然到达：
女朋友："能听到！"（又建立了一个连接）
你：？？？我没说话啊
结果：女朋友一直等，资源浪费

三次握手防止历史重复连接
```

## IP：找到去上海的路

### 例子：寄快递的地址系统

```
IP 地址 = 省 + 市 + 区 + 门牌号

192.168.1.100
┌─────┐┌─────┐┌─────┐┌─────┐
│ 192 ││ 168 ││  1  ││ 100 │
└─────┘└─────┘└─────┘└─────┘
  网络号              主机号

路由 = 问路 + 转发

北京路由器：
"去 192.168.x.x？往南走，给天津"

天津路由器：
"去 192.168.x.x？继续往南，给济南"

... 层层转发，直到到达

每个路由器只关心下一跳，不关心完整路径
路径可以动态变化，哪条路通走哪条
```

## 实际案例

### 案例 1：打开网页时发生了什么？

```
你在浏览器输入：example.com

1. DNS 查询（应用层）
   "example.com 的 IP 是多少？"
   ↓
   返回：93.184.216.34

2. 建立 TCP 连接（传输层）
   你的电脑 ←──三次握手──→ 93.184.216.34
   ↓
   连接建立

3. 发送 HTTP 请求（应用层）
   "GET / HTTP/1.1"
   ↓
   TCP 分割成多个包，编号发送

4. 网络传输（网络层 + 链路层）
   你的电脑 → 路由器 → ISP → 骨干网 → 目标服务器
   ↓
   可能丢包、乱序、出错

5. TCP 重组（传输层）
   接收所有包，按序号排序，校验完整性
   ↓
   发现丢包 #3 → 请求重发
   收到 #3 → 确认

6. HTTP 响应组装（应用层）
   把数据交给浏览器渲染
   ↓
   你看到网页

总时间：几百毫秒（北京到美国）
```

### 案例 2：为什么视频通话会卡顿？

```
正常情况：
你的视频帧 ──→ 网络 ──→ 对方的屏幕
延迟 50ms，流畅

卡顿情况1：网络拥塞
你的视频帧 ──→ 网络堵车 ──→ 延迟 500ms
对方看到：你 0.5 秒前的画面

卡顿情况2：丢包重传
你的视频帧 #100 丢了
TCP：必须重传，等收到 #100 才能显示
对方看到：画面卡住，等 #100 到了才继续

解决方案：视频通话用 UDP，不是 TCP
UDP：
- 不保证到达
- 丢了就丢了，继续播下一帧
- 结果：偶尔花屏，但不卡顿

权衡：
- 文件传输用 TCP（必须完整）
- 视频通话用 UDP（实时优先）
```

### 案例 3：为什么下载速度会波动？

```
下载一个大文件：

开始（慢启动）：
速度：100 KB/s
      ↓
200 KB/s
      ↓
400 KB/s
      ↓
800 KB/s（发现丢包！）
      ↓
400 KB/s（减半）
      ↓
500 KB/s（线性增长）
      ↓
600 KB/s
      ↓
700 KB/s（又丢包！）
      ↓
350 KB/s（再减半）
      ↓
...

结果：速度在"试探上限"和"丢包减速"之间波动
最终稳定在网络能承载的速度
```

## 自测问题

**问：TCP 怎么保证数据不丢？**

答案：
- 每发一个包，等对方确认（ACK）
- 超时没收到确认，重发
- 对方收到重复包，丢弃但再发一次确认（防 ACK 丢失）

**问：TCP 怎么保证顺序正确？**

答案：
- 每个包有编号（序号）
- 接收方按序号排序
- 先到的后序号，先存着，等前面的到了再一起处理

**问：为什么视频通话用 UDP，文件传输用 TCP？**

答案：
- TCP：保证完整，但可能卡顿（等重传）
- UDP：不保证到达，但不卡顿（丢了就跳过）
- 文件必须完整，视频可以偶尔花屏

**问：三次握手为什么需要三次？**

答案：
- 防止历史重复连接请求造成混乱
- 确认双方的发送和接收能力都正常

## 总结

网络协议就是电脑的"快递系统"：

| 现实问题 | 快递解决 | TCP/IP 解决 |
|---------|---------|------------|
| 损坏 | 包装、验货 | 校验和 |
| 丢失 | 单号追踪、补发 | 确认应答、重传 |
| 顺序乱 | 贴标签排序 | 序号 |
| 对方收不下 | 分批送货 | 滑动窗口 |
| 路堵了 | 换条路 | 拥塞控制、动态路由 |

分层让每个部分简单可维护，就像快递公司分部门。
