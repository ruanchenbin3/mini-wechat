# 网络协议栈：从物理传输到可靠通信

核心问题：电信号传输不可靠，如何在不可靠的物理层上构建可靠的通信？

---

## 物理层的现实：信号传输不可靠

### 电信号传输中的问题

```
发送方发送比特流：101101001011...
           ↓
传输介质（铜线/光纤/无线）：
- 信号衰减：电压降低，0/1 界限模糊
- 电磁干扰：外部电磁波改变信号
- 多径效应：无线信号反射，波形叠加失真
- 时钟漂移：发送方和接收方时钟不同步
           ↓
接收方可能收到：101101101011...（某位翻转）
```

**物理层只能尽力传输比特，不保证正确性**。

## 链路层：局域网内的可靠传输

### 以太网帧结构

```
┌────────┬────────┬────────┬────────┬────────┬────────┐
│ 前导码  │ 目的MAC │ 源MAC  │  类型   │  数据   │  CRC   │
│ 8字节  │ 6字节  │ 6字节  │ 2字节  │46-1500B│ 4字节  │
└────────┴────────┴────────┴────────┴────────┴────────┘

CRC（循环冗余校验）：
- 发送方计算数据的 CRC 值，附在帧尾
- 接收方重新计算，与帧尾 CRC 对比
- 不一致则丢弃该帧

MAC 地址：
- 48 位硬件地址，全球唯一
- 用于局域网内寻址
```

### 链路层的局限

```
局域网 A                    局域网 B
┌─────────┐                ┌─────────┐
│ 主机 A1 │                │ 主机 B1 │
│ 主机 A2 │◄─────路由器────►│ 主机 B2 │
│ 主机 A3 │                │ 主机 B3 │
└─────────┘                └─────────┘
  MAC 寻址                    MAC 寻址

问题：
- MAC 地址只在局域网内有效
- 跨网络通信需要新的寻址机制
- 链路层不保证端到端可靠（中间路由器可能丢包）
```

## 网络层：跨网络的寻址与路由

### IP 协议的核心功能

```
IP 地址结构（IPv4）：
┌─────────────────────┬─────────────────────┐
│      网络号          │       主机号         │
│    192.168.1        │        .100         │
└─────────────────────┴─────────────────────┘

子网掩码 255.255.255.0 区分网络号和主机号

IP 数据报格式：
┌────────┬────────┬────────┬────────┬────────┬────────┐
│ 版本/IHL│ 服务类型│ 总长度  │ 标识   │标志/偏移│ TTL   │
│  协议   │ 首部校验│  源IP   │ 目的IP │  选项   │  数据  │
└────────┴────────┴────────┴────────┴────────┴────────┘

TTL（Time To Live）：
- 每经过一个路由器减 1
- 到 0 时丢弃，防止路由环路
```

### 路由选择

```
路由表（简化）：
┌─────────────────┬─────────────────┬───────────┐
│   目标网络       │     下一跳       │   接口    │
├─────────────────┼─────────────────┼───────────┤
│   192.168.1.0   │     直接投递     │   eth0    │
│   10.0.0.0      │    192.168.1.1   │   eth0    │
│   0.0.0.0       │    192.168.1.1   │   eth0    │
└─────────────────┴─────────────────┴───────────┘

最长前缀匹配：
- 目标 192.168.1.100 → 匹配 192.168.1.0/24
- 目标 8.8.8.8 → 匹配 0.0.0.0/0（默认路由）

动态路由协议（OSPF/BGP）：
- 自动发现网络拓扑变化
- 计算最短路径
- 故障时自动切换
```

### IP 的不可靠性

```
IP 提供"尽力而为"服务：
- 不保证数据报到达
- 不保证按序到达
- 不保证不重复
- 不保证不损坏

原因：
- 路由器缓冲区满时丢弃数据报
- 不同路径导致乱序
- 网络拥塞导致丢包

需要上层协议（TCP）提供可靠性
```

## 传输层：端到端的可靠通信

### TCP 解决的核心问题

| 问题 | 解决方案 | 机制 |
|-----|---------|------|
| 数据损坏 | 错误检测 | 校验和 |
| 丢包 | 重传机制 | 确认应答 + 超时重传 |
| 乱序 | 顺序控制 | 序号 + 确认号 |
| 接收方过载 | 流量控制 | 滑动窗口 |
| 网络拥塞 | 拥塞控制 | 慢启动、拥塞避免 |

### TCP 段结构

```
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ 源端口 │ 目的端口│   序号   │   确认号  │数据偏移│  标志  │  窗口   │
│  16bit │  16bit │  32bit  │  32bit   │ 4bit  │ 6bit  │  16bit │
├────────┴────────┴────────┴────────┴────────┴────────┴────────┴────────┤
│ 校验和  │ 紧急指针│              选项（可变）              │  数据   │
│ 16bit  │ 16bit  │                                      │        │
└────────┴────────┴──────────────────────────────────────┴────────┘

关键字段：
- 序号（Sequence Number）：本报文段数据的第一个字节序号
- 确认号（Acknowledgment Number）：期望收到的下一个字节序号
- 窗口（Window）：接收方缓存空间大小，用于流量控制
- 标志（Flags）：SYN、ACK、FIN、RST 等控制位
```

### 三次握手建立连接

```
客户端                              服务器
   │                                  │
   │  SYN=1, seq=x                   │  ← 客户端请求建立连接
   │─────────────────────────────────►│     初始序号 x
   │                                  │
   │  SYN=1, ACK=1, seq=y, ack=x+1   │  ← 服务器同意建立连接
   │◄─────────────────────────────────│     初始序号 y，确认 x+1
   │                                  │
   │  ACK=1, seq=x+1, ack=y+1        │  ← 客户端确认
   │─────────────────────────────────►│
   │                                  │
   │         连接建立，开始传输数据      │

为什么是三次？
- 防止历史重复连接请求（延迟的 SYN）造成错误
- 同步双方的初始序号（ISN）
- 确认双方的发送和接收能力正常
```

### 滑动窗口与流量控制

```
发送方窗口：
┌─────────────────────────────────────────┐
│ 已确认 │ 已发送未确认 │ 可发送 │ 不可发送 │
│ 1-100 │   101-400   │401-800 │  801+   │
└─────────────────────────────────────────┘
         ↑              ↑
       窗口前沿        窗口后沿

接收方通告窗口（rwnd）：
- TCP 首部中的 Window 字段
- 表示接收方缓存剩余空间
- 发送方根据 rwnd 调整发送速率

动态调整：
- 接收方处理快 → rwnd 增大 → 发送方加速
- 接收方处理慢 → rwnd 减小 → 发送方减速
- rwnd = 0 时，发送方停止发送（零窗口探测）
```

### 拥塞控制

```
拥塞窗口（cwnd）：发送方根据网络状况调整的窗口

慢启动（Slow Start）：
初始：cwnd = 1 MSS（最大段大小）
每收到一个 ACK：cwnd += 1 MSS
每 RTT：cwnd 翻倍（指数增长）

拥塞避免（Congestion Avoidance）：
当 cwnd > ssthresh（慢启动阈值）：
每 RTT：cwnd += 1 MSS（线性增长）

快速重传与快速恢复：
收到 3 个重复 ACK → 认为丢包
- ssthresh = cwnd / 2
- cwnd = ssthresh + 3 MSS
- 重传丢失段，进入拥塞避免

超时重传：
RTO（重传超时）到期未收到 ACK
- ssthresh = cwnd / 2
- cwnd = 1 MSS
- 重新慢启动
```

### TCP 与 UDP 的对比

| 特性 | TCP | UDP |
|-----|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠传输 | 尽力而为 |
| 顺序 | 保证有序 | 不保证 |
| 流量控制 | 有 | 无 |
| 拥塞控制 | 有 | 无 |
| 头部开销 | 20 字节 | 8 字节 |
| 适用场景 | 文件传输、HTTP | 视频流、DNS、游戏 |

## 应用层：具体应用的协议

### HTTP/HTTPS

```
请求报文：
GET /index.html HTTP/1.1\r\n
Host: www.example.com\r\n
User-Agent: Mozilla/5.0\r\n
Accept: text/html\r\n
\r\n

响应报文：
HTTP/1.1 200 OK\r\n
Content-Type: text/html\r\n
Content-Length: 1234\r\n
\r\n
<html>...</html>

HTTPS = HTTP + TLS/SSL：
- 对称加密传输数据
- 非对称加密交换密钥
- 数字证书验证身份
```

### DNS

```
域名解析过程：

用户查询：www.example.com
           ↓
本地 DNS 缓存
           ↓
递归查询：
  根服务器（.）→ com 服务器
           ↓
  com 服务器 → example.com 服务器
           ↓
  example.com 服务器 → 192.0.2.1
           ↓
返回 IP 地址

DNS 记录类型：
- A：IPv4 地址
- AAAA：IPv6 地址
- CNAME：别名
- MX：邮件服务器
- NS：域名服务器
```

## 数据封装过程

```
应用层数据（HTTP 请求）
           ↓
传输层（TCP）：
  加 TCP 首部（端口、序号、确认号、窗口）
           ↓
网络层（IP）：
  加 IP 首部（源/目的 IP、TTL、协议类型）
           ↓
链路层（以太网）：
  加以太网首部（MAC 地址、类型）
  加 CRC 尾部
           ↓
物理层：
  转换为电信号/光信号发送

接收方逐层解封装，去掉首部，交给上层
```

### 观察 TCP 连接状态

```bash
# 查看所有 TCP 连接
ss -tan

# 输出：
# State      Recv-Q  Send-Q  Local Address:Port  Peer Address:Port
# ESTAB      0       0       192.168.1.100:54321  93.184.216.34:443
# TIME-WAIT  0       0       192.168.1.100:54320  93.184.216.34:443

状态说明：
- ESTAB：连接已建立
- SYN-SENT：已发 SYN，等待确认
- TIME-WAIT：连接已关闭，等待 2MSL 确保对方收到 ACK
```

sudo tcpdump -i any host example.com -nn -S

## 实际案例分析

### 案例 1：为什么 TCP 建立连接需要三次握手？

```
场景：网络中存在延迟的重复 SYN

客户端发送 SYN（seq=x），网络延迟滞留
客户端重发 SYN（seq=x'），建立连接，传输数据，关闭连接

滞留的 SYN 突然到达服务器
服务器回复 SYN-ACK（seq=y, ack=x+1）
客户端已关闭，不响应
服务器一直等待，资源浪费（半开连接攻击）

三次握手防止：
- 客户端收到重复的 SYN-ACK 会发送 RST 终止连接
- 或者客户端不响应，服务器超时释放资源
```

### 案例 2：TCP 队头阻塞

```
HTTP/2 多路复用在一个 TCP 连接上：

流 1：请求 index.html ■■■■■■■■■■■■■■■■
流 2：请求 style.css  ■■■■
流 3：请求 script.js  ■■■■■■■■

如果流 1 的一个包丢失：
流 1：■■■■■■□■■■■■■■■■  ← 丢包，等待重传
流 2：■■■■               ← 已到达，但不能交付
流 3：■■■■■■■■           ← 已到达，但不能交付

虽然流 2、3 的数据已完整到达，但必须等流 1 重传后才能按序交付

HTTP/3 使用 QUIC（基于 UDP）解决：
每个流独立传输，互不影响
```

### 案例 3：TIME_WAIT 状态的作用

```
连接关闭的四次挥手：

客户端                    服务器
   │                        │
   │  FIN, seq=u           │  ← 客户端：我要关闭
   │───────────────────────►│
   │                        │
   │  ACK, ack=u+1         │  ← 服务器：知道了
   │◄───────────────────────│
   │                        │
   │  FIN, seq=w, ack=u+1  │  ← 服务器：我也要关闭
   │◄───────────────────────│
   │                        │
   │  ACK, ack=w+1         │  ← 客户端：知道了
   │───────────────────────►│
   │                        │
   │    客户端进入 TIME_WAIT（2MSL） │

TIME_WAIT 作用：
1. 确保最后一个 ACK 到达服务器（如果丢失，服务器重发 FIN）
2. 防止延迟的数据包被新连接误收（相同四元组的新连接）

2MSL = 2 * Maximum Segment Lifetime（通常 2-4 分钟）
```

## 总结

网络协议栈的核心思想：**分层解耦，逐层增强**

| 层次 | 功能 | 关键协议/机制 |
|-----|------|-------------|
| 应用层 | 具体应用逻辑 | HTTP、DNS、TLS |
| 传输层 | 端到端可靠通信 | TCP（连接、可靠、有序）、UDP（快速）|
| 网络层 | 跨网络寻址路由 | IP、路由协议、TTL |
| 链路层 | 局域网可靠传输 | 以太网、MAC、CRC |
| 物理层 | 比特传输 | 电信号、光信号、无线 |

每一层：
- 为上层提供服务
- 使用下层的服务
- 隐藏实现细节
- 独立演进
