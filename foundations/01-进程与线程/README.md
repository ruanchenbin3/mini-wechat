## 1. 从物理视角看"程序"

### 程序是什么？

想象你写了一个实验步骤文档：

```
实验：测量重力加速度

步骤1：准备单摆装置
步骤2：测量摆长 L
步骤3：测量周期 T
步骤4：计算 g = 4π²L/T²
步骤5：记录结果
```

这个文档就是**程序**——**静态的、存在硬盘上的指令序列**。

它本身不会做任何事情，就像实验手册不会自动做实验。

### 进程是什么？

当你**开始执行**这个实验：
- 你走进实验室（分配了空间）
- 你拿出器材（加载了资源）
- 你按照步骤操作（执行指令）
- 你在记录本上写数据（产生输出）

这就是**进程**——**动态的、正在运行的程序实例**。

### 核心类比

| 概念 | 实验类比 | 计算机对应 |
|-----|---------|-----------|
| 程序 | 实验手册（静态文档） | 硬盘上的可执行文件 |
| 进程 | 正在进行的实验（动态活动） | 内存中的运行实例 |
| 实验室 | 实验进行的空间 | 内存空间 |
| 实验器材 | 实验需要的工具 | CPU、内存、文件句柄 |
| 实验记录本 | 实验过程中产生的数据 | 进程的数据区 |

## 2. 进程的物理组成

一个进程在内存中是什么样子的？

```
┌─────────────────────────────────────┐ 高地址
│           栈区 (Stack)               │  ← 局部变量、函数调用信息
│              ↓ 向下增长               │     像草稿纸，用完就扔
├─────────────────────────────────────┤
│                                     │
│           堆区 (Heap)                │  ← 动态分配的数据
│              ↑ 向上增长               │     像储物柜，需要管理
│                                     │
├─────────────────────────────────────┤
│         数据段 (Data)                │  ← 全局变量、静态变量
│                                     │     实验的固定配置
├─────────────────────────────────────┤
│         代码段 (Code)                │  ← 程序指令
│                                     │     实验步骤文档的副本
└─────────────────────────────────────┘ 低地址
```

### 代码段（Code Segment）

**是什么：** 程序的机器指令

**物理意义：** 就像你把实验手册复印了一份放在实验台上，随时查看

**特点：**
- 只读（防止程序意外修改自己的指令）
- 多个进程可以共享同一份代码（节省内存）

### 数据段（Data Segment）

**是什么：** 全局变量和静态变量

**物理意义：** 实验的固定配置，比如"重力加速度取 9.8"

**例子：**
```c
int global_var = 100;  // 数据段
static int static_var = 200;  // 数据段

void function() {
    int local_var = 300;  // 不在数据段，在栈区
}
```

### 堆区（Heap）

**是什么：** 动态分配的内存

**物理意义：** 实验过程中临时需要的储物空间

**例子：**
```c
// 申请一块内存（像打开一个储物柜）
int* arr = malloc(100 * sizeof(int));

// 使用这块内存
arr[0] = 1;
arr[1] = 2;

// 释放内存（像关闭储物柜）
free(arr);
```

**关键问题：** 储物柜需要手动管理，忘了关就是**内存泄漏**。

### 栈区（Stack）

**是什么：** 函数调用和局部变量

**物理意义：** 实验过程中的草稿纸

**例子：**
```c
void experiment() {
    int length = 100;      // 栈上：测量长度
    double period = 2.0;   // 栈上：测量周期
    
    double g = calculate(length, period);  // 函数调用入栈
    
    // 函数结束，这些数据自动销毁
}
```

**特点：**
- 自动管理（像草稿纸，用完自动翻页）
- 空间有限（栈溢出 = 草稿纸用完了）
- 后进先出 LIFO（像叠盘子）

## 3. 为什么需要线程？

### 单线程的问题

想象你一个人做实验：
- 你在等水烧开（I/O 操作）
- 这段时间你**什么都做不了**，只能干等
- 水烧开后，你才能继续下一步

这就是**单线程**——一个进程只有一个执行流，遇到等待就只能阻塞。

### 多线程的解决方案

现在你有两个实验台：
- 实验台1：等水烧开（I/O）
- 实验台2：同时准备器材（计算）

**注意：** 还是你一个人在做，但你在两个任务间**快速切换**。

这就是**多线程**——一个进程内有多个执行流，可以并发执行。

### 线程的物理组成

线程共享进程的地址空间，但有自己的：
- **栈**（自己的草稿纸）
- **寄存器状态**（做到哪一步了）
- **程序计数器**（下一条指令在哪里）

```
进程（实验室）
├─ 线程1（实验台1）
│   ├─ 自己的栈
│   └─ 自己的寄存器
│
├─ 线程2（实验台2）
│   ├─ 自己的栈
│   └─ 自己的寄存器
│
└─ 共享区域
    ├─ 代码段（共同的实验手册）
    ├─ 数据段（共同的配置）
    └─ 堆区（共同的储物柜）
```

### 线程的问题：竞争条件

两个实验台同时用一个天平（共享资源）：
- 实验台1：把样品放上去，准备读数
- 实验台2：突然把自己的样品放上去
- 实验台1：读到了错误的数值

这就是**竞争条件**——多个线程同时访问共享数据，结果不确定。

**解决方案：** 加锁（Mutex）
- 用天平前先"上锁"
- 用完再"解锁"
- 其他人看到锁就等待

但锁也有问题：**死锁**（两个人互相等对方释放资源）。

## 4. 并发 vs 并行

### 并发（Concurrency）

**定义：** 多个任务在**同一时间段内**推进，不一定同时执行。

**物理类比：** 一个人做多个实验，快速切换。

**特点：**
- 单核 CPU 就能实现
- 通过快速切换（时间片轮转）模拟"同时进行"
- 提高资源利用率（一个任务等待时做另一个）

### 并行（Parallelism）

**定义：** 多个任务**同时**执行。

**物理类比：** 多个人同时做不同的实验。

**特点：**
- 需要多核 CPU
- 真正的同时执行
- 提高吞吐量

### 图示

```
单核 CPU 的并发：
时间 →
CPU1: [任务A][任务B][任务A][任务B][任务A][任务B]
      （快速切换，看起来同时进行）

多核 CPU 的并行：
时间 →
CPU1: [任务A][任务A][任务A]
CPU2: [任务B][任务B][任务B]
      （真正同时进行）
```

## 5. 为什么 Node.js 选择单线程？

### 传统多线程的问题

每个线程需要：
- 独立的栈空间（通常 1-8 MB）
- 寄存器状态保存
- 线程切换的上下文开销

**物理类比：** 每个实验台都要配一套完整的器材，切换实验台还要收拾/重新布置。

如果有 10000 个连接，就需要 10000 个线程——内存爆炸。

### Node.js 的方案：事件驱动 + 非阻塞 I/O

**核心思想：** 一个线程处理所有任务，但遇到 I/O 不等待。

```
Node.js 单线程模型：

[事件循环] ← 调度任务
    ↓
[任务1: 读取文件] → 交给操作系统 → 立即返回，继续下一个
    ↓
[任务2: 处理数据] → CPU 计算 → 完成
    ↓
[任务3: 发送请求] → 交给操作系统 → 立即返回，继续下一个
    ↓
...
    ↓
[文件读取完成] ← 操作系统通知 ← 事件循环调度回调
```

**物理类比：**
- 你一个人管理 100 个实验台
- 每个实验台需要等待时（水烧开），你不去等，而是去看其他实验台
- 水烧开后，铃声响起（事件通知），你再回来处理

### 优势

1. **内存占用小** —— 一个线程 vs N 个线程
2. **没有锁的问题** —— 单线程没有竞争条件
3. **适合 I/O 密集型** —— 即时通讯主要是网络 I/O

### 劣势

1. **不适合 CPU 密集型** —— 一个任务卡住，所有任务都卡住
2. **不能利用多核** —— 需要开多个进程

## 6. 动手验证

### 观察进程

```bash
# 查看当前运行的进程
ps aux

# 查看进程树
pstree

# 查看某个进程的线程
ps -T -p <进程ID>
```

### 简单实验：创建进程

```javascript
// process-demo.js
console.log('进程ID:', process.pid);
console.log('父进程ID:', process.ppid);

// 查看内存使用
console.log('内存使用:', process.memoryUsage());

// 查看 CPU 使用（当前进程）
console.log('CPU 使用:', process.cpuUsage());
```

运行：`node process-demo.js`

### 简单实验：创建子进程

```javascript
// spawn-demo.js
const { spawn } = require('child_process');

console.log('主进程 PID:', process.pid);

// 创建子进程（像开一个新的实验室）
const child = spawn('node', ['-e', 'console.log("子进程 PID:", process.pid)']);

child.stdout.on('data', (data) => {
  console.log(data.toString());
});
```

## 7. 学习总结

### 核心概念

| 概念 | 一句话解释 | 物理类比 |
|-----|-----------|---------|
| 程序 | 静态的指令集合 | 实验手册 |
| 进程 | 运行的程序实例 | 正在进行的实验 |
| 线程 | 进程内的执行流 | 实验台 |
| 并发 | 时间段内多任务推进 | 一个人快速切换 |
| 并行 | 同时多任务执行 | 多个人同时做 |

### 关键理解

1. **进程是资源分配的单位** —— 每个进程有独立的内存空间
2. **线程是调度的单位** —— CPU 实际执行的是线程
3. **多线程共享内存** —— 高效但需处理竞争
4. **Node.js 单线程事件驱动** —— 用异步 I/O 避免阻塞

### 与即时通讯的关系

- 每个用户连接对应一个 Socket
- Node.js 用单线程管理成千上万个 Socket
- 消息到来是"事件"，通过事件循环处理
- 不需要为每个连接创建一个线程
