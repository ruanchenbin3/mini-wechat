# 文件系统与数据库：从易失内存到持久存储

核心问题：内存断电即失，如何在慢速、非易失的磁盘上高效、可靠地存储数据？

---

## 物理约束：存储介质的矛盾

### 速度与持久性的权衡

| 存储介质 | 访问延迟 | 持久性 | 成本/GB | 适用场景 |
|---------|---------|--------|---------|---------|
| DRAM（内存）| 100 ns | 断电即失 | $10 | 运行时数据 |
| SSD（闪存）| 100 μs | 断电保持 | $0.1 | 热数据、系统盘 |
| HDD（磁盘）| 10 ms | 断电保持 | $0.02 | 大容量冷数据 |
| 磁带 | 秒级 | 断电保持 | $0.001 | 归档备份 |

**核心矛盾**：
- 快的（DRAM）不持久
- 持久的（磁盘）慢 10^5 倍
- 需要权衡：什么数据放内存，什么放磁盘

## 噩梦一：磁盘访问太慢

### 机械硬盘的物理限制

```
磁盘结构：
┌─────────────────────────────────────────┐
│  盘片（磁性涂层）                        │
│  ┌─────────┐                            │
│  │  磁道    │ ← 同心圆                   │
│  │   扇区   │ ← 最小读写单位，通常 512B   │
│  └─────────┘                            │
│       ↑                                  │
│    磁头（悬浮在盘片上方）                 │
└─────────────────────────────────────────┘

访问时间组成：
1. 寻道时间：磁头移动到正确磁道（平均 4-10 ms）
2. 旋转延迟：盘片旋转到正确扇区（平均 2-4 ms）
3. 传输时间：实际读写数据（约 100 MB/s）

随机访问 4KB：
- 寻道 5 ms + 旋转 2 ms + 传输 0.04 ms ≈ 7 ms
- 比内存访问慢 70,000 倍

顺序访问 4KB：
- 无需寻道，直接传输 0.04 ms
- 比随机访问快 175 倍
```

### 文件系统的优化：局部性

```
思想：把一起访问的数据放在一起

顺序写入：
文件 A 的数据块：1 → 2 → 3 → 4 → 5（连续）
读取时：磁头一次定位，顺序读取

随机写入：
文件 A 的数据块：1 → 50 → 3 → 200 → 5（分散）
读取时：每次都要寻道

预读（Read-ahead）：
读取块 1 时，预测接下来要读块 2、3
提前加载到内存，减少磁盘访问
```

## 噩梦二：断电导致数据丢失或损坏

### 场景：正在写入文件时断电

```
用户操作：保存文档

正常流程：
1. 写入数据到磁盘缓存
2. 更新文件元数据（大小、修改时间）
3. 更新目录项
4. 同步到磁盘

断电发生在第 2 步后：
- 数据已写入
- 但元数据没更新
- 结果：文件存在，但大小为 0，数据丢失

断电发生在第 3 步后：
- 数据和元数据已写入
- 但目录没更新
- 结果：文件存在，但目录找不到它
```

### 解决方案：日志（Journal）

```
传统方式（非日志文件系统）：
直接修改磁盘上的数据结构
断电后可能处于不一致状态
需要 fsck 扫描修复（慢，可能丢数据）

日志文件系统（ext4、NTFS、APFS）：
先写日志，再写实际数据

日志内容：
┌─────────────────────────────────────────┐
│ 事务开始                                │
│ 操作1：分配数据块 100 给文件 A           │
│ 操作2：更新文件 A 的 inode（大小=4KB）   │
│ 操作3：更新目录项                        │
│ 事务结束                                │
└─────────────────────────────────────────┘

写入流程：
1. 把日志写入磁盘（顺序写，快）
2. 把实际数据写入磁盘（随机写，慢）
3. 标记事务完成

断电恢复：
- 检查未完成的日志事务
- 重新执行（重做）或撤销（回滚）
- 保证文件系统一致性

代价：
- 写操作变成两次（日志 + 实际数据）
- 但日志是顺序写，开销相对小
```

## 噩梦三：大容量存储如何快速查找

### 线性扫描的问题

```
100 万文件，线性查找：
平均需要扫描 50 万个文件名
每次磁盘访问 10 ms
总时间：50 万 × 10 ms = 5000 秒 ≈ 1.4 小时

不可接受。
```

### 解决方案：B+ 树索引

```
B+ 树结构（以文件名为键）：

                    [file10, file20, file30]
                   /         |          \
        [file1, file5]  [file15, file18]  [file25, file28]
           /    |    \      /     |     \     /     |     \
        数据块 数据块 数据块 数据块 数据块 数据块 数据块 数据块 数据块

查找 file18：
1. 根节点：file18 > file10, < file20 → 中间分支
2. 中间节点：file18 > file15 → 右边
3. 叶子节点：找到 file18 → 获取数据块地址

磁盘访问次数 = 树高（通常 3-4 层）
查找 100 万文件：3-4 次磁盘访问 ≈ 30-40 ms

B+ 树特性：
- 所有数据在叶子节点，叶子节点链表连接（范围查询快）
- 节点大小 = 磁盘块大小（通常 4KB），一次读一个节点
- 自平衡，插入删除自动调整
```

## 噩梦四：小文件浪费空间

### 块大小与空间浪费

```
磁盘以块为单位分配（通常 4KB）：

文件 A：100 字节
分配：1 个块（4KB）
浪费：4096 - 100 = 3996 字节（97.5%）

文件 B：5000 字节
分配：2 个块（8KB）
浪费：8192 - 5000 = 3192 字节（39%）

1000 个 100 字节的小文件：
实际数据：100 KB
占用空间：1000 × 4KB = 4 MB
浪费：97.5%
```

### 解决方案：块内碎片管理

```
ext4 的解决方案：

1. 块组（Block Group）
   把磁盘分成多个块组，每个块组有独立的 inode 表和数据块
   相关文件放在同一块组，减少磁头移动

2. 延迟分配（Delayed Allocation）
   不立即分配物理块，先缓存写入操作
   等数据积累到一定量，一次性分配连续块
   提高顺序性，减少碎片

3. 内联数据（Inline Data）
   小文件直接存储在 inode 中
   无需额外数据块

4. 稀疏文件（Sparse File）
   只分配实际写入的数据块
   空洞部分不占用磁盘空间
```

## 数据库：在文件系统之上

### 为什么需要数据库？

```
文件系统的局限：
- 无结构化查询（只能按路径访问）
- 无事务支持（并发修改可能冲突）
- 无索引（大量数据查找慢）

数据库提供的功能：
- SQL 查询语言
- ACID 事务
- B+ 树索引
- 缓存管理
```

### 数据库页管理

```
数据库以页（Page）为单位管理数据（通常 8KB 或 16KB）：

页类型：
- 数据页：存储表记录
- 索引页：存储 B+ 树节点
- 溢出页：存储大字段（TEXT、BLOB）
- 系统页：元数据、事务日志

页结构：
┌─────────────────────────────────────────┐
│ 页头（Page Header）                      │
│ - 页号、页类型、记录数、空闲空间指针      │
├─────────────────────────────────────────┤
│ 记录目录（Record Directory）             │
│ - 每条记录在页内的偏移位置                │
├─────────────────────────────────────────┤
│ 记录数据（Records）                      │
│ - 实际存储的数据行                        │
├─────────────────────────────────────────┤
│ 空闲空间（Free Space）                   │
└─────────────────────────────────────────┘

页内查找：
- 记录按主键有序排列
- 页内二分查找（内存操作，快）
- 跨页通过 B+ 树索引
```

### 缓冲池（Buffer Pool）

```
思想：把常用数据缓存到内存

缓冲池结构：
┌─────────────────────────────────────────┐
│ 缓冲池（内存，GB 级）                     │
│ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐        │
│ │页 1 │ │页 5 │ │页 3 │ │页 8 │ ...     │
│ │热数据│ │热数据│ │热数据│ │热数据│        │
│ └─────┘ └─────┘ └─────┘ └─────┘        │
│ 最近最少使用（LRU）替换策略               │
└─────────────────────────────────────────┘
           ↓ 未命中时从磁盘加载
┌─────────────────────────────────────────┐
│ 磁盘（TB 级）                            │
│ 所有数据页                               │
└─────────────────────────────────────────┘

访问数据：
1. 检查缓冲池（内存访问，100 ns）
2. 命中：直接返回
3. 未命中：从磁盘加载（10 ms），加入缓冲池

命中率通常 > 95%，平均访问时间接近内存速度
```

### 事务与 ACID

```
事务：一组操作，要么全成功，要么全失败

ACID 特性：
A - Atomicity（原子性）
   事务是不可分割的最小执行单位
   全部成功提交，或全部回滚

C - Consistency（一致性）
   事务执行前后，数据库处于一致状态
   约束、触发器都满足

I - Isolation（隔离性）
   并发事务互不干扰
   通过锁或 MVCC 实现

D - Durability（持久性）
   事务提交后，数据永久保存
   即使断电也不丢失

实现机制：
- 预写日志（WAL）：先写日志，再写数据
- 锁：行锁、表锁，控制并发访问
- MVCC：多版本并发控制，读不阻塞写
```

### LSM 树：写优化的数据结构

```
传统 B+ 树的问题：
- 随机写：更新一条记录可能需要修改多个磁盘页
- 写放大：一次逻辑写变成多次物理写

LSM 树（Log-Structured Merge Tree）思想：
- 写操作只追加到日志，不修改原数据
- 定期合并（Compaction），把多个小文件合并成大文件

结构：
┌─────────────────────────────────────────┐
│ MemTable（内存，有序结构）                │
│ 写入先到这里，达到一定大小后刷盘           │
├─────────────────────────────────────────┤
│ SSTable Level 0（磁盘，多个小文件）       │
│ 从 MemTable 刷下来的数据                  │
│ 文件内有序，文件间可能重叠                  │
├─────────────────────────────────────────┤
│ SSTable Level 1（磁盘，更大的文件）       │
│ Level 0 合并而来，文件间不重叠              │
├─────────────────────────────────────────┤
│ SSTable Level 2+（磁盘，越来越大）        │
│ 逐层合并，每层大小是上一层的 10 倍          │
└─────────────────────────────────────────┘

读取：
1. 先查 MemTable
2. 再查 Level 0（从新到旧）
3. 再查 Level 1、2...
4. 找到就返回

优点：
- 写是顺序追加，快
- 适合写密集型场景（日志、时序数据）

缺点：
- 读可能需要查多个文件
- 合并时消耗磁盘带宽

代表：LevelDB、RocksDB、Cassandra、HBase
```

## 案例分析

### 案例 1：为什么数据库插入变慢了？

现象：MySQL 表数据量增大后，插入速度明显下降。

分析：
```
小表时：
- 所有索引在内存
- 插入只需修改内存中的 B+ 树
- 速度快

大表时：
- 索引太大，无法全部缓存
- 插入需要修改 B+ 树，可能触发磁盘 I/O
- 维护索引的成本增加

解决方案：
- 批量插入（减少日志写入次数）
- 删除不必要的索引
- 分区表（把大表分成多个小表）
```

### 案例 2：SSD 为什么比 HDD 快？

```
HDD（机械硬盘）：
- 寻道时间：4-10 ms（磁头移动）
- 旋转延迟：2-4 ms（盘片旋转）
- 随机访问：约 10 ms

SSD（固态硬盘）：
- 无机械部件
- 电子寻址：约 100 μs
- 随机访问比 HDD 快 100 倍

但 SSD 有写入放大问题：
- 闪存必须先擦除再写入
- 擦除块（256KB）> 写入页（4KB）
- 修改 4KB 可能需要擦除重写 256KB
-  wear leveling（磨损均衡）算法缓解
```

### 案例 3：为什么 Redis 快，MySQL 慢？

```
Redis（内存数据库）：
- 数据全在内存
- 访问延迟：100 ns
- 单线程，无锁竞争
- 适合：缓存、会话、实时计数

MySQL（磁盘数据库）：
- 数据主要在磁盘
- 访问延迟：10 ms（未命中缓冲池）
- 多线程，需要锁机制
- 适合：持久化存储、复杂查询、事务

不是替代关系，是互补：
应用 → Redis（缓存）→ 命中？直接返回
              ↓ 未命中
         MySQL（持久存储）→ 返回数据
              ↓
         更新 Redis 缓存
```

## 自测问题

**问：为什么磁盘顺序访问比随机访问快这么多？**

答案：
- 机械硬盘需要寻道和旋转，随机访问每次都要 10 ms
- 顺序访问只需一次定位，然后连续传输
- 差距：175 倍（HDD），SSD 差距较小但仍存在

**问：日志文件系统如何保证断电不损坏？**

答案：
- 先写日志（顺序写，快）
- 再写实际数据（随机写，慢）
- 断电后检查未完成的日志事务
- 重做或回滚，保证一致性

**问：数据库缓冲池的作用是什么？**

答案：
- 缓存热数据页在内存
- 减少磁盘 I/O
- 命中率通常 > 95%
- 用 LRU 算法管理替换

**问：LSM 树适合什么场景？**

答案：
- 写密集型（日志、时序数据）
- 写是顺序追加，快
- 读可能需要查多个文件，适合写多读少

## 总结

文件系统和数据库解决的核心问题：**易失内存 vs 持久存储的速度和可靠性矛盾**

关键机制：
1. **局部性**：顺序访问、预读，减少磁盘 I/O
2. **日志**：先写日志再写数据，保证断电安全
3. **索引**：B+ 树、LSM 树，加速查找
4. **缓冲**：热数据缓存到内存，弥合速度差距
5. **事务**：WAL、锁、MVCC，保证 ACID

```
速度层次：
CPU 寄存器：0.3 ns
L1 缓存：    1 ns
内存：       100 ns
SSD：        100 μs
HDD：        10 ms
网络：       150 ms

每层差 100-1000 倍，缓存和局部性是关键优化手段
```
