# 虚拟内存：让每个程序都以为自己独占整个内存

核心问题：物理内存有限，但程序需要更多地址空间，如何让每个程序都觉得自己有无限的内存？

---

## 物理约束：内存很贵，程序很贪婪

想象你是一位房东（操作系统），只有 **100 平米** 的房子（物理内存），但有 **10 个租客**（程序），每个都声称需要 **100 平米**。

总需求：1000 平米  
实际拥有：100 平米  

怎么办？

---

## 核心洞察：程序不会同时用到所有空间

观察发现：

| 现象 | 说明 |
|-----|------|
| 程序启动时 | 只加载了代码的一部分 |
| 运行时 | 只访问当前需要的数据 |
| 大部分时间 | 80% 的代码从未被执行（错误处理、罕见功能） |
| 数据访问 | 集中在少数几个热点区域 |

**关键发现**：程序虽然"声称"需要 100 平米，但"实际使用"的可能只有 10-20 平米。

---

## 解决方案：虚拟内存

给每个程序一张"地图"（虚拟地址空间），让它以为自己有 100 平米，但实际上只分配真正使用的部分。

```
程序 A 的视角：
┌─────────────────────────────────────┐
│  虚拟地址空间（4GB）                 │
│  0x00000000 - 0xFFFFFFFF            │
│  ┌─────────┐  代码段                │
│  │ 实际在用 │  ← 映射到物理内存      │
│  ├─────────┤                        │
│  │ 还没用到 │  ← 暂时不分配          │
│  ├─────────┤                        │
│  │ 实际在用 │  ← 映射到物理内存      │
│  │ 数据段   │                        │
│  ├─────────┤                        │
│  │ 还没用到 │  ← 暂时不分配          │
│  │ ...     │                        │
│  └─────────┘                        │
└─────────────────────────────────────┘
           ↓ 地址翻译（页表）
┌─────────────────────────────────────┐
│  物理内存（实际只有 8GB）            │
│  ┌─────┐ ┌─────┐ ┌─────┐           │
│  │ A的 │ │ B的 │ │ C的 │  ← 实际分配 │
│  │代码 │ │代码 │ │数据 │            │
│  └─────┘ └─────┘ └─────┘           │
│  （只占用了程序声称的一小部分）       │
└─────────────────────────────────────┘
```

---

## 页表：虚拟地址到物理地址的地图

虚拟内存按 **页（Page）** 管理，通常是 **4KB**。

```
虚拟地址 0x12345678 的翻译过程：

┌─────────────────────────────────────┐
│  虚拟地址：0x12345678               │
│  ┌───────────┬───────────┐          │
│  │  页号     │  页内偏移  │          │
│  │ 0x12345   │  0x678    │          │
│  └─────┬─────┴───────────┘          │
│        ↓                            │
│  查页表 → 物理页框号 0x00ABC         │
│        ↓                            │
│  物理地址 = 0x00ABC * 4096 + 0x678   │
│           = 0x00ABC678              │
└─────────────────────────────────────┘
```

**页表**就是一张大表，记录每个虚拟页对应哪个物理页框。

---

## 多级页表：节省内存的 trick

问题：32 位系统，4GB 地址空间，4KB 页大小
- 需要 4GB / 4KB = 1M 个页表项
- 每个页表项 4 字节
- 页表本身就要 4MB！

如果 100 个程序同时运行，页表就要 400MB，比很多程序本身还大。

**解决方案：多级页表**

```
单级页表（浪费）：
┌─────────────────────────────────────┐
│ 页表项 0      → 物理页 0            │
│ 页表项 1      → 物理页 1            │
│ ...（全部 1M 项，大部分未使用）      │
│ 页表项 1048575 → 物理页 xxx         │
└─────────────────────────────────────┘

二级页表（节省）：
┌─────────────────────────────────────┐
│ 页目录（1024 项）                    │
│ ├─ 项 0 → 指向页表 0（未分配，NULL）│
│ ├─ 项 1 → 指向页表 1（已分配）      │
│ │         ├─ 页表项 0 → 物理页      │
│ │         └─ ...                    │
│ ├─ 项 2 → 未分配（NULL）            │
│ └─ ...（大部分目录项都是 NULL）      │
└─────────────────────────────────────┘
```

**关键**：程序只用了虚拟地址空间的一小部分，大部分页目录项是空的，不需要分配二级页表。

实际节省：从 4MB 降到几十 KB。

## 页面置换：内存满了怎么办？

当物理内存用完，又有新页面需要加载时：

```
物理内存已满：
┌─────────────────────────────────────┐
│ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐    │
│ │ A的 │ │ B的 │ │ C的 │ │ D的 │    │
│ │页面 │ │页面 │ │页面 │ │页面 │    │
│ │  1  │ │  2  │ │  3  │ │  4  │    │
│ └─────┘ └─────┘ └─────┘ └─────┘    │
└─────────────────────────────────────┘

E 程序需要加载新页面：
1. 选一个"牺牲者"页面（比如 B 的页面 2）
2. 如果 B 的页面 2 被修改过，写回磁盘（交换区）
3. 把 E 的新页面装入腾出的位置
4. 更新页表
```

**选谁当牺牲者？** —— 页面置换算法

| 算法 | 思想 | 问题 |
|-----|------|------|
| FIFO | 先进先出 | 可能淘汰常用页面 |
| LRU | 最近最少使用 | 实现复杂，需要记录访问时间 |
| Clock | 近似 LRU，用访问位 | 平衡效果和开销 |

实际系统大多用 **Clock 算法** 或其变体。

## 写时复制（Copy-on-Write）：fork 的魔法

问题：`fork()` 创建子进程时，需要复制父进程的所有内存吗？

```
传统 fork：
父进程 4GB 内存 → 复制 4GB → 子进程
耗时：复制 4GB 数据！

写时复制 fork：
父进程 4GB 内存 ──┐
                  ├── 共享同一份物理页，标记为"只读"
子进程 4GB 虚拟 ──┘

当父或子修改某页时：
1. 触发页错误（只读保护）
2. 操作系统复制该页
3. 各自拥有独立副本
4. 继续执行

结果：只复制真正修改的页面（通常很少）
```

**效果**：`fork()` 从"复制 4GB"变成"复制几 KB 的页表"，速度提升 1000 倍。

## 内存映射文件（mmap）：磁盘文件变内存

```
传统文件读取：
┌─────────┐    read()    ┌─────────┐    复制    ┌─────────┐
│ 磁盘文件 │ ───────────→ │ 内核缓冲区 │ ─────────→ │ 用户内存 │
│ data.bin │              │（页缓存）  │            │  buf[]  │
└─────────┘              └─────────┘            └─────────┘
                          数据复制了两次！

mmap 方式：
┌─────────┐    mmap()    ┌─────────┐
│ 磁盘文件 │ ───────────→ │ 虚拟地址  │
│ data.bin │              │ 空间     │
└─────────┘              └────┬────┘
                              │
                    ┌─────────┴─────────┐
                    │  按需加载页面      │
                    │  访问哪页读哪页    │
                    └───────────────────┘
```

**优势**：
- 没有 `read()` 的系统调用开销
- 没有内核到用户空间的复制
- 大文件可以只加载需要的部分
- 多个进程可以共享同一份物理页（共享内存）

## 缺页中断：按需加载的实现机制

```
程序访问虚拟地址 0x12345000：

1. CPU 查页表，发现该页"不存在"
           ↓
2. 触发缺页中断（Page Fault）
           ↓
3. 操作系统接管：
   a. 找一块空闲物理页（或置换）
   b. 从磁盘加载该页到物理内存
   c. 更新页表：虚拟页 → 物理页
   d. 恢复程序执行
           ↓
4. 程序继续，仿佛什么都没发生
           （只是这次访问慢了 ~10ms）
```

**关键**：程序启动时，大部分页面都是"不存在"状态，第一次访问时才加载。

这就是"按需分页"（Demand Paging）。

## 虚拟内存的核心收益

| 收益 | 说明 |
|-----|------|
 **内存超额订阅** | 10 个程序各需 4GB，物理只有 8GB，都能运行 |
| **隔离保护** | 程序 A 无法访问程序 B 的内存（页表隔离） |
| **简化编程** | 每个程序从 0 开始编址，不用关心物理位置 |
| **共享内存** | 多个程序映射同一物理页，高效共享数据 |
| **按需加载** | 只加载需要的代码/数据，启动更快 |

---

## 关键数字

| 项目 | 大小 | 说明 |
|-----|------|------|
| 典型页面大小 | 4 KB | x86 默认，ARM64 也常用 |
| 大页（Huge Page） | 2 MB / 1 GB | 减少 TLB miss，数据库常用 |
| 32 位虚拟地址空间 | 4 GB | 0x00000000 - 0xFFFFFFFF |
| 64 位虚拟地址空间 | 256 TB | 实际可用，理论更大 |
| 页表遍历时间 | 几十 ns | 硬件 MMU 加速 |
| TLB miss 惩罚 | 几十 ns | 需要查页表 |
| 缺页中断处理 | 10 ms | 从磁盘加载 |

## 实验观察

### 查看进程的虚拟内存布局

```bash
# Linux
pmap <PID>
# 或
cat /proc/<PID>/maps

# macOS
vmmap <PID>
```

观察：
- 代码段（text）、数据段（data）、堆（heap）、栈（stack）的分布
- 大量地址范围是空的（未分配）

### 观察缺页中断

```bash
# Linux
ps -o min_flt,maj_flt <PID>
# min_flt: 轻微缺页（已加载到内存，只需映射）
# maj_flt: 严重缺页（需要从磁盘加载）
```

### 观察内存使用

```bash
# Linux
top -p <PID>
# VIRT: 虚拟内存大小
# RES:  实际物理内存（Resident）
# SHR:  共享内存

# 通常 VIRT >> RES，这就是虚拟内存的超额订阅
```

## 案例分析

### 案例 1：Chrome 为什么吃内存？

现象：打开 10 个标签页，Chrome 占用 2GB+ 内存。

分析：
- 每个标签页是一个进程（隔离性）
- 每个进程有独立的虚拟地址空间
- 但许多页面是共享的（Chromium 代码、共享库）
- `top` 显示的内存是虚拟内存总和，实际物理内存没那么多

验证：
```bash
# 查看 Chrome 进程的 RSS vs VSIZE
ps aux | grep chrome
# RSS 列是物理内存，VSZ 是虚拟内存
# 通常 VSZ 是 RSS 的 2-5 倍
```

### 案例 2：Swap 满了系统为什么卡死？

现象：内存不足时，系统开始使用 Swap（磁盘交换区），然后变得极其卡顿。

原因分析：
- 正常缺页：从磁盘加载代码/数据，10 ms，可接受
- Swap 缺页：把内存页换出到磁盘，再把需要的页换入
- 如果工作集大于物理内存，频繁换入换出（抖动，Thrashing）
- 每次访问都可能触发磁盘 I/O，系统忙于换页，无法执行程序

解决方案：
- 加内存
- 或者减少同时运行的程序
- 或者使用 `cgroups` 限制进程内存

### 案例 3：Docker 容器的内存限制

现象：Docker 设置了 `--memory=512m`，但程序申请 1GB 内存没有立即报错。

原因：
- 虚拟内存可以超额申请（`malloc` 不立即分配物理页）
- 只有真正访问时才分配
- 当实际使用超过 512m 时，容器被 OOM kill

验证：
```bash
# 在容器内
docker run --memory=512m -it ubuntu
# 运行一个不断分配内存的程序
# 观察什么时候被 kill
```

## 自测问题

**问：虚拟内存和物理内存的区别是什么？**

答案：
- 物理内存 = 真实的 RAM 芯片，容量有限
- 虚拟内存 = 每个进程的"幻觉"地址空间，可以比物理内存大得多
- 通过页表映射，虚拟地址翻译成物理地址

**问：为什么需要多级页表？单级有什么问题？**

答案：
- 单级页表需要为整个虚拟地址空间建立映射，即使大部分未使用
- 32 位系统单级页表就要 4MB
- 多级页表按需分配，未使用的区域不占用页表空间
- 实际节省 90%+ 的页表内存

**问：TLB 是什么？为什么重要？**

答案：
- TLB（Translation Lookaside Buffer）= 页表的高速缓存
- 每次内存访问都要查页表，太慢
- TLB 缓存最近的地址翻译，命中时几乎无开销
- TLB miss 需要遍历页表，慢几十倍

**问：写时复制有什么好处？**

答案：
- `fork()` 时避免复制大量内存
- 父子进程共享物理页，直到有人修改
- 只复制修改的页面，节省内存和时间
- 是 Linux 能高效创建进程的关键

## 总结

虚拟内存解决的核心问题：**有限的物理内存 vs 无限的地址空间需求**

关键机制：
1. **页表** —— 虚拟地址到物理地址的映射
2. **按需分页** —— 用缺页中断实现懒加载
3. **页面置换** —— 内存满时选择牺牲者
4. **写时复制** —— 延迟复制，节省内存

记住这个逻辑链：

```
物理约束：内存贵且有限，程序贪婪且不会同时用所有内存
    ↓
解决方案：虚拟内存 —— 给每个程序一张"地图"
    ↓
新问题：页表太大
    ↓
解决方案：多级页表 —— 按需分配
    ↓
新问题：内存满了怎么办
    ↓
解决方案：页面置换 —— 把不用的页换到磁盘
    ↓
新问题：fork 复制太慢
    ↓
解决方案：写时复制 —— 共享直到修改
```

每个机制都是在特定约束下的工程优化。
