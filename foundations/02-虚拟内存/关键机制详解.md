# 虚拟内存关键机制详解

本文是对主文档中四个关键机制的补充说明，用更具体的例子帮助理解。

---

## 1. 页表：地址翻译

### 核心问题
程序用虚拟地址（如 0x00401234），但内存芯片只认物理地址。怎么转换？

### 解决方案：页表 = 翻译字典

```
程序说："我要访问 0x00401234"
           ↓
CPU 查页表（字典）：
  虚拟页 0x00401 → 物理页 0x12
           ↓
实际访问物理地址 0x00012234
```

### 为什么需要页表？

**没有页表（直接访问物理内存）**：
- 程序 A 编译时用地址 0x1000
- 程序 B 编译时也用地址 0x1000
- 两个程序不能同时运行，冲突！

**有页表（虚拟内存）**：
- 程序 A 以为自己用 0x1000，实际映射到物理 0x5000
- 程序 B 以为自己用 0x1000，实际映射到物理 0x8000
- 两个程序同时运行，互不干扰

## 2. 缺页中断：按需加载

### 核心问题
程序启动时声称需要 1GB，但实际只用到 100MB。如果一开始就加载 1GB，浪费 900MB。

### 解决方案：用到才加载

```
程序启动：
┌─────────────────────────────────────┐
│ 虚拟地址空间（1GB）                  │
│ ┌─────────┐ ┌─────────┐            │
│ │ 代码段  │ │ 数据段  │            │
│ │（未加载）│ │（未加载）│            │
│ └─────────┘ └─────────┘            │
└─────────────────────────────────────┘
  页表标记：这些页"不存在"

程序执行，访问代码段：
           ↓
CPU：这页不存在！触发缺页中断
           ↓
操作系统：从磁盘加载代码到内存
           ↓
更新页表：现在存在了
           ↓
程序继续，仿佛什么都没发生
```

### 具体例子

**打开微信**：
- 微信程序 500MB
- 启动时只加载：
  - 登录界面代码（5MB）
  - 基础库（10MB）
- 其他 485MB 未加载：
  - 聊天功能代码（用到再加载）
  - 朋友圈代码（用到再加载）
  - 小程序代码（用到再加载）

**效果**：启动快（2 秒 vs 20 秒），内存省（15MB vs 500MB）。

## 3. 页面置换：内存扩展

### 核心问题
物理内存只有 8GB，但 10 个程序各需要 1GB。第 9、10 个程序怎么运行？

### 解决方案：把不用的页换到磁盘

```
物理内存（8GB）已满：
┌─────────────────────────────────────┐
│ A │ B │ C │ D │ E │ F │ G │ H │    │
└─────────────────────────────────────┘

程序 I 想启动：
1. 看哪些页很久没访问（比如 A 的某些页）
2. 把这些页写到磁盘（Swap）
3. 腾出的空间给 I 用
4. 如果 A 后来又访问这些页，从磁盘读回来
```

### 形象比喻

**书桌只有 1 平米，但你有 10 平米的书**：
- 把正在看的书放在桌上（物理内存）
- 把不看的书放回书架（磁盘 Swap）
- 需要时再取回来

### 关键数字

| 操作 | 时间 | 对比 |
|-----|------|------|
| 访问内存 | 100 ns | 基准 |
| 缺页中断（从磁盘加载）| 10 ms | 慢 10 万倍 |

**代价**：如果频繁换入换出（抖动），系统会卡死。

## 4. 写时复制：高效 fork

### 核心问题
创建子进程时，必须复制父进程所有内存吗？4GB 复制要几秒，太慢。

### 解决方案：先共享，修改时再复制

---

### 场景一：Shell 执行命令

```bash
$ ls
```

背后发生了什么？

```
1. Shell 进程（父进程，4GB 内存）
   ┌─────────────────────────────────────┐
   │ 代码 │ 数据 │ 环境变量 │ 历史记录   │
   └─────────────────────────────────────┘

2. Shell 调用 fork() 创建子进程
   - 子进程要执行 ls 命令
   - 但 fork 后子进程和父进程一模一样

3. 子进程调用 exec("ls") 
   - 把内存换成 ls 程序的内存
   - 执行 ls
```

#### 传统 fork：全部复制（慢）

```
父进程 4GB ──→ 复制 4GB ──→ 子进程 4GB
              （耗时 3 秒）

然后 exec("ls")：
子进程 4GB ──→ 扔掉 ──→ 加载 ls（几百 KB）

结果：复制了 4GB，只用了 0 秒，全部浪费！
```

#### 写时复制 fork：先共享，修改再复制

```
fork() 时：
父进程 4GB ──┐
             ├──→ 同一份物理内存（标记"只读"）
子进程 4GB ──┘

没有复制！只是增加了引用计数。
耗时：1 毫秒

┌─────────────────────────────────────┐
│ 物理内存中的同一份数据               │
│ 代码 │ 数据 │ 环境变量 │ 历史记录   │
│ （父和子共享，只读）                │
└─────────────────────────────────────┘
```

然后 exec("ls")：
```
子进程扔掉共享的内存，加载 ls
父进程继续用自己的内存

全程没有复制任何页！
```

---

### 场景二：子进程真的需要修改数据

```python
# Python 解释器 fork 子进程处理请求
```

```
父进程（Python）：
┌─────────────────────────────────────┐
│ 变量 x = 100                        │
│ 变量 y = 200                        │
└─────────────────────────────────────┘

fork() 后，父子共享同一份内存（只读）

子进程想修改 x = 999：
           ↓
发现内存是只读的，触发页错误
           ↓
操作系统介入：
1. 复制包含 x 的那一页（4KB）
2. 子进程拿到自己的副本
3. 在副本上修改 x = 999
4. 父进程的 x 仍然是 100

结果：
- 只复制了 1 页（4KB），不是 4GB
- 其他未修改的页仍然共享
```

---

### 形象比喻：复印文件

| 方式 | 做法 | 问题 |
|-----|------|------|
| **传统 fork** | 先复印 1000 页，再看要不要改 | 复印了 1000 页，可能只改 1 页 |
| **写时复制** | 两人看同一份原件，谁要改哪页再复印哪页 | 只复印真正修改的页 |

---

### 核心收益

```
传统 fork：
- 复制 4GB 内存
- 耗时：3 秒
- 内存：翻倍（8GB）

写时复制 fork：
- 复制 0 页
- 耗时：1 毫秒
- 内存：不变（4GB）
- 之后只复制修改的页（通常 < 1MB）
```

**为什么能这样做？**
- 大多数 fork 后很快 exec，根本不修改内存
- 即使修改，也只改少数几页
- 复制整份内存是极大的浪费

---

## 一句话总结

| 机制 | 解决什么问题 | 核心思想 |
|-----|-----------|---------|
| **页表** | 虚拟地址→物理地址 | 查字典翻译 |
| **缺页中断** | 启动慢、内存浪费 | 用到才加载 |
| **页面置换** | 物理内存不够 | 不用的放磁盘 |
| **写时复制** | fork 太慢 | 先共享，修改再复制 |

## 实验观察

### 观察写时复制的效果

```bash
# Linux
# 查看进程的内存映射
pmap -x <PID>

# 关注：
# - Anonymous pages: 私有内存（已复制）
# - Shared pages: 共享内存（写时复制未触发）
```

### 观察 fork 的速度

```c
// test_fork.c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    // 分配 1GB 内存
    char *data = malloc(1024 * 1024 * 1024);
    memset(data, 0, 1024 * 1024 * 1024);
    
    printf("Before fork\n");
    
    pid_t pid = fork();
    if (pid == 0) {
        // 子进程
        printf("Child\n");
        exit(0);
    } else {
        // 父进程
        wait(NULL);
        printf("Parent\n");
    }
    
    return 0;
}
```

```bash
$ time ./test_fork
Before fork
Child
Parent

real    0m0.002s  # 只有 2 毫秒！如果复制 1GB 需要几百毫秒
```
